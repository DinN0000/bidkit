# SA (Solution Architecture) Domain Context

SA 팀이 SSOT를 작성할 때 적용하는 구조적 패턴과 품질 기준.

## 팀 구성과 역할

SA 팀 내 4명의 에이전트가 릴레이로 작업한다:

| 역할 | 담당 |
|------|------|
| **Team Lead** | 솔루션 선정 방향 지시, 비교 전략, 결과 발표 |
| **Researcher** | 제품 스펙, 인증서, 레퍼런스, 가격 정보 조사 |
| **Writer** | 5-Part 솔루션 소개, 비교표, 라이선스 분석 초안 |
| **Critic** | 성능 측정조건, 인증 유효기간, 비교표 공정성, 라이선스-비용 정합성 검증 |

### SA 팀 산출물

| 산출물 | 핵심 구조 규칙 |
|--------|-------------|
| 제품별 솔루션 스펙 | **5-Part**: 개요 → 특징 → 아키텍처 → 인증/레퍼런스 → 라인업 |
| 제품 비교표 | 공정성 원칙 (자사 유리 기준만 선정 금지) |
| 레퍼런스 케이스 | 고객사+프로젝트명+기간+규모, 3년 이내 우선 |
| 라이선스 분석 | 유형+과금단위+범위+유지보수율, **비용표 정합성 필수** |

## 솔루션 소개 표준 구조

모든 솔루션/제품 소개 섹션은 5-Part로 작성한다:

| 순서 | 파트 | 포함 내용 |
|------|------|---------|
| 1 | **개요** | 제품명, 제조사, 한 줄 포지셔닝, 시장 위치 |
| 2 | **주요 특징** | 3~5개 bullet, 각각 정량적 근거 포함 |
| 3 | **아키텍처** | 구성도 1장 + 계층별 역할 설명 |
| 4 | **인증 및 레퍼런스** | 인증서 목록(번호+유효기간) + 적용 사례 표 |
| 5 | **제품 라인업** | 모델별 스펙 비교표, 제안 모델 명시 |

파트 순서를 지키되, 해당 솔루션에 맞지 않는 파트는 생략 가능 (예: 오픈소스는 라인업 대신 에디션 비교).

## 성능 수치 표기 규칙

**모든 성능 수치에는 측정 조건을 반드시 명시한다.**

```
좋은 예: "초당 3,000 TPS (Fabric 2.2, 4-peer, LevelDB, 서명 미포함)"
나쁜 예: "초당 3,000 TPS"
나쁜 예: "높은 처리 성능을 제공합니다"
```

조건 필수 항목:
- 제품/플랫폼 버전
- 하드웨어 또는 노드 구성
- 측정 환경 (운영/테스트/벤치마크)
- 포함/제외 조건 (암호화, 서명, 네트워크 지연 등)

## 제품 비교표 패턴

경쟁 제품 비교 시 다음 차원을 포함한다:

| 비교 차원 | 설명 | 주의사항 |
|---------|------|---------|
| 기반 기술 | 프로토콜, 프레임워크, 아키텍처 | 기술 세대 차이 명시 |
| 표준 준수 | W3C, ISO, FIPS, 국내 인증 | 인증 범위(버전/모델) 특정 |
| 핵심 기능 | 차별점 3~5개 | 양사에 공정한 기준 선정 |
| 적용 사례 | 국내외 레퍼런스 | 유사 규모/업종 우선 |
| 라이선스 | 오픈소스/상용/하이브리드 | 총 비용(TCO) 관점 포함 |

**공정성 원칙**: 자사에 유리한 기준만 선정하지 않는다. Critic은 비교 차원의 공정성을 검증한다.

## 레퍼런스 케이스 작성 패턴

| 항목 | 필수 | 설명 |
|------|------|------|
| 고객사 | ✓ | 공개 가능 시 실명, 불가 시 "국내 5대 시중은행" 수준 |
| 프로젝트명 | ✓ | 공식 사업명 |
| 적용 솔루션 | ✓ | 제안 솔루션과의 연관성 명시 |
| 기간/규모 | ✓ | 계약 기간, 투입 인력 또는 사업 규모 |
| 정량 성과 | 선택 | "처리량 40% 향상", "장애 건수 연 0건" 등 |

케이스는 **최근 3년 이내** 우선. 3년 초과 시 현재 운영 상태(Active/마이그레이션 완료 등) 명시.

## 라이선스 분석

솔루션 제안 시 라이선스 구조를 반드시 명시:

| 구분 | 내용 |
|------|------|
| 유형 | 영구/구독/오픈소스 |
| 과금 단위 | 코어/노드/사용자/트랜잭션 기반 |
| 포함 범위 | 기본 기능, 추가 모듈, 기술 지원 레벨 |
| 유지보수 | 연간 요율 (보통 라이선스의 15~22%) |

**비용표와의 정합성**: SA에서 명시한 라이선스 수량이 비용표와 불일치하면 Critical 이슈.

## 아키텍처 다이어그램 규칙

1. **계층 분리**: 사용자 → 어플리케이션 → 플랫폼 → 인프라
2. **SDK/API 표시**: 연동 지점에 SDK/API 유형 레이블 명시
3. **프로토콜 표시**: 연동 화살표에 HTTP, gRPC, TCP/IP 등 명시
4. **보안 경계**: 망 구분(DMZ, 내부망)이 있다면 시각적으로 구분
5. **범례**: 컬러나 선 스타일로 구분 시 반드시 범례 포함

## 팀 간 의존성

SA 산출물의 의존 관계:

```
BA ──────────────▶ SA  (요구사항 → 솔루션 식별)
SA ──────────┬──▶ TA  (H/W 요구사항 → 인프라)
             └──▶ TA  (라이선스 → 비용)
```

- BA의 요구사항에서 필요한 솔루션을 식별한다
- SA에서 결정된 H/W 요구사항(예: HSM 장비)은 TA 인프라에 반영된다
- SA의 라이선스 수량과 비용은 TA 비용표에 반영된다

### 권한 체계

| 데이터 유형 | 권한 |
|-----------|------|
| 제품 스펙, 인증, 모델번호 | **SA** |
| 요구사항, 프로세스 | BA |
| 인프라 수량, 서버 사이징 | TA |
| 데이터 모델, 마이그레이션 | DA |

비권한 SSOT가 권한 SSOT와 충돌하면, 비권한 쪽이 수정된다.

## Critic 검증 포인트 (SA)

| 검증 항목 | Critical 조건 |
|---------|-------------|
| 성능 조건 누락 | 수치에 측정 조건 없음 |
| 인증 유효기간 | 만료된 인증을 유효한 것처럼 기술 |
| 비교표 공정성 | 자사 유리 기준만 선정 |
| 버전 호환성 | 제안 솔루션 간 검증되지 않은 버전 조합 |
| 레퍼런스 최신성 | 5년 초과 사례만 존재, 현재 상태 미기재 |
| 라이선스-비용 불일치 | SA 수량 ≠ 비용표 수량 |
