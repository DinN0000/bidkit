# BA (Business Analysis) Domain Context

BA 팀이 SSOT를 작성할 때 적용하는 구조적 패턴과 품질 기준.

---

## 업무 기능 서술의 2단 구조

**모든 업무 기능은 "흐름도 → 화면"의 2단 구조로 작성한다.** 이것이 탑티어 제안서와 평범한 제안서를 가르는 핵심 패턴.

### 1단: 업무 흐름도

| 요소 | 위치 | 내용 |
|------|------|------|
| 참여자(Actor) | 좌측 세로 배치 | 사용자, 기관, 시스템 등 |
| 프로세스 | 중앙 | 처리 단계, 판단, 분기 |
| 상세 흐름 | 우측 사이드 | 번호별 상세 설명 (① ② ③) |

상세 흐름에는 단순 나열이 아니라 **"왜"와 "어떻게"**를 포함:
```
① 처리 요청
  - 사용자가 앱에서 요청 전송
  - 요청 시 본인인증 선행 필요
② 검증 수행
  - 잔액 확인 및 한도 체크
  - 부적격 시 사유 코드와 함께 반려
```

### 2단: 화면 설계

```
┌────────────────────┬─────────────────┐
│                    │  주요 기능        │
│  [화면 캡처/목업]   │                  │
│                    │ • 기능 설명 1     │
│                    │ • 기능 설명 2     │
│                    │ • 기능 설명 3     │
└────────────────────┴─────────────────┘
```

화면 좌측에 캡처/와이어프레임, 우측에 **주요 기능** 사이드바. 이 레이아웃을 제안서 전체에서 일관되게 유지.

## 업무 분류 체계

업무는 **참여자(Actor) 기준**으로 분류한다:

| 분류 기준 | 예시 역할 | 서술 관점 |
|---------|---------|---------|
| 관리/운영 기관 | 중앙 관리자, 감독기관 | 정책 설정, 승인, 모니터링 |
| 참여/중계 기관 | 은행, 중계사, 파트너 | 업무 처리, 분배, 연동 |
| 최종 사용자 | 개인, 기업 고객 | 서비스 이용 (가입/조회/거래) |
| 수납/제공자 | 가맹점, 서비스 제공자 | 수납, 정산, 서비스 운영 |
| 시스템 관리자 | 내부 운영자 | 시스템 관리, 설정, 통계 |

같은 기능도 참여자가 다르면 별도 섹션으로 작성 (예: "관리자의 사용자 관리" vs "사용자의 프로필 관리").

## 업무별 분량 가이드

| 업무 복잡도 | 예상 분량 | 포함 내용 |
|----------|---------|---------|
| 단순 (조회/확인) | 1~2p | 흐름도 + 화면 1개 |
| 보통 (등록/수정) | 2~3p | 흐름도 + 화면 2~3개 |
| 복잡 (다자간 거래/승인) | 3~5p | 흐름도 + 시퀀스 + 화면 3~5개 |
| 핵심 프로세스 | 5~10p | 개요 + 흐름도 + 시퀀스 + 화면 + 예외처리 |

핵심 업무에 분량이 부족하면 제안서 전체 인상이 약해진다. Critic은 분량 균형을 검증.

---

## 시퀀스 다이어그램 규칙

복잡한 다자간 프로세스는 시퀀스 다이어그램으로 표현:

### 참여자 배치
- **좌측**: 요청자 (사용자, 클라이언트 앱)
- **중앙**: 서비스 서버, 비즈니스 로직
- **우측**: 외부 시스템, 인증/검증, 저장소

### 넘버링
- 메인 흐름: 1, 2, 3, ...
- 조건부 분기: 3-1, 3-2
- 하위 흐름이 길어지면 별도 다이어그램으로 분리 (예: "상세(1/3)", "상세(2/3)")

### 분량 기준
시퀀스가 30 step을 초과하면 분리. 한 다이어그램에 과도한 step은 가독성을 떨어뜨린다.

## 업무별/기능별 구현 방안 목차 구조

BA의 핵심 산출물인 "업무별/기능별 구현 방안"은 다음 계층으로 작성:

```
개발 방안
├── 1. 어플리케이션 아키텍처 (개요)
├── 2. 업무별/기능별 구현 방안
│   ├── 1) 대분류 업무 A
│   │   ├── 가) 소분류 업무 A-1
│   │   ├── 나) 소분류 업무 A-2
│   │   └── 다) 소분류 업무 A-3
│   ├── 2) 대분류 업무 B
│   │   └── ...
│   └── 3) 대분류 업무 C
│       └── ...
├── 3. 테스트 방안 (→ TA 참조)
└── 4. 이행 방안 (→ TA 참조)
```

대분류는 RFP 요구사항의 **상위 업무 단위**, 소분류는 **개별 기능 단위**.

## 차별점 서술 패턴

제안서에서 기술적 차별점을 강조할 때:

### 2-Column 비교 구조

```
┌──────────────────────────┬──────────────────────────┐
│ 일반적 SI 접근 방식         │ 제안사의 차별적 접근 방식    │
├──────────────────────────┼──────────────────────────┤
│ • 표준 방법론 적용          │ • 도메인 특화 방법론 적용    │
│ • 범용 프로세스             │ • 검증된 핵심 활동 Ready-Made│
└──────────────────────────┴──────────────────────────┘
```

### 차별점 서술 3요소

1. **주장**: 무엇이 차별적인가
2. **근거**: 왜 차별적인가 (수행 실적, 특허, 인증)
3. **고객 효과**: 고객에게 어떤 구체적 이점이 있는가

근거 없는 차별점은 Critic이 Warning 처리.

## 사용자 인수 테스트 서술

BA에서 사용자 인수 테스트를 기술할 때:

### 5단계 수행 절차

| 단계 | 프로젝트 | 고객 |
|------|---------|------|
| 분석 | 기준 도출 | 계획서 확인/승인 |
| 개발 | 시나리오 작성 | 시나리오 검토 |
| 테스트 | 케이스 작성 | 테스트 수행 |
| 이행 | 결과 보고서 | 실행 |
| 종료 | 확인서 공유 | 최종 승인 |

**고객 참여 시점**을 반드시 명시. "고객이 뭘 해야 하는지"가 빠지면 실행 가능성이 의심받는다.

## Critic 검증 포인트 (BA)

| 검증 항목 | Critical 조건 |
|---------|-------------|
| RFP 커버리지 | RFP 요구사항 중 업무 흐름에 매핑 안 된 항목 존재 |
| 참여자 누락 | 업무 흐름에 관련 Actor 미포함 |
| 2단 구조 미준수 | 업무 기능에 흐름도 또는 화면이 빠짐 |
| 번호 불연속 | 시퀀스 번호 건너뜀 또는 중복 |
| 분량 불균형 | 핵심 업무보다 부수 업무에 분량이 더 많음 |
| 예외 처리 누락 | 오류/실패/거부 시나리오 미기술 |
| 용어 불일치 | 같은 업무를 다른 이름으로 호칭 |
| 화면-흐름 단절 | 흐름도 단계와 화면이 매핑되지 않음 |
