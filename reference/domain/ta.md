# TA (Tech Architecture) Domain Context

TA 팀이 SSOT를 작성할 때 적용하는 구조적 패턴과 품질 기준.

## 팀 구성과 역할

TA 팀 내 4명의 에이전트가 릴레이로 작업한다:

| 역할 | 담당 |
|------|------|
| **Team Lead** | 인프라 설계 방향 지시, 비용 조율, 결과 발표 |
| **Researcher** | 서버/장비 스펙, 가격, 벤더 정보, 인증 조사 |
| **Writer** | 아키텍처 다이어그램, H/W·S/W 구성표, 이행 계획, 비용표 초안 |
| **Critic** | 수량-비용 정합성, HA 구성, 망분리 준수, 임계값 정의 검증 |

### TA 팀 산출물

| 산출물 | 핵심 구조 규칙 |
|--------|-------------|
| 시스템 아키텍처 | 네트워크, 보안, 3-Layer 접근제어 |
| H/W·S/W 구성 | **3환경 분리**, **수량 = 비용표 일치** |
| 이행 계획 | **3원칙**: 사전준비, 리허설, 롤백 |
| 비용 산정 | H/W, S/W, 라이선스, 유지보수 |
| 테스트 방안 | 단위 → 통합 → 성능 → 인수 → 병행 |

## H/W·S/W 구성표

금융 IT 제안서에서 인프라 구성은 **환경별 분리**가 기본이다.

### 환경 분리 원칙

| 환경 | 용도 | 가용성 요건 |
|------|------|---------|
| **운영** | 실서비스 | HA 필수 (Active-Standby 또는 Active-Active) |
| **개발** | 개발 및 단위 테스트 | 단일 구성 가능 |
| **검증(테스트)** | 통합/성능/인수 테스트 | 운영 환경 미러링 권장 |

RFP에서 환경 분리를 명시하지 않더라도, 금융권이면 3환경 분리를 기본 제안.

### H/W 구성표 표준 컬럼

```
| 구분 | 용도 | 모델 | CPU | 메모리 | 디스크 | 수량 | 비고 |
```

### S/W 구성표 표준 컬럼

```
| 구분 | 제품명 | 버전 | 용도 | 라이선스 유형 | 수량 |
```

**수량 일치 규칙**: H/W 수량 × S/W 수량이 비용표와 정확히 일치해야 한다. 이 불일치는 금융 제안서에서 가장 흔한 Critical 이슈.

## 네트워크 및 접근제어

### 3-Layer 접근제어

| Layer | 내용 | 표현 방식 |
|-------|------|---------|
| 권한 분리 | 사용자→역할→권한→대상 매핑 | 매핑표 |
| 접근제어 | 주체별 인증 방식 + 접근 대상 | 접근제어 매트릭스 |
| 네트워크 분리 | 대외망/DMZ/내부망 구분 | 네트워크 구성도 |

### 인터페이스 정의 패턴

시스템 간 연동은 다음 항목을 빠짐없이 기술:

```
| 연동 구간 | 프로토콜 | 데이터 포맷 | 보안 | 방향 |
```

- 모든 대외 연동에 암호화 방식 명시
- FEP/EAI/API Gateway 등 미들웨어 경유 시 명시
- 연동 실패 시 대응(재시도, 큐잉, 알림) 포함

## 플랫폼 구성도

### 표준 구조

```
개발환경: IDE → 형상관리 → CI/CD → Registry
    │ 배포
    ▼
운영환경: SDK → 어플리케이션 → 미들웨어 → 인프라
         관리도구: 모니터링/관리 콘솔
```

개발환경과 운영환경은 반드시 물리적/논리적 분리를 표현.

### 모니터링 필수 항목

| 대상 | 지표 | 임계값 정의 필요 여부 |
|------|------|-----------------|
| 서버 상태 | CPU, 메모리, 디스크 | ✓ |
| 어플리케이션 | 응답시간, 에러율 | ✓ |
| 네트워크 | 대역폭, 패킷 손실 | ✓ |
| 데이터 | 저장소 사용률, 동기화 상태 | ✓ |

## 테스트 방법론

### 테스트 유형 매트릭스

| 유형 | 범위 | 산출물 | 환경 | 참여자 |
|------|------|--------|------|--------|
| 단위 | 모듈/함수 | 테스트 결과서 | 개발 | 개발자 |
| 통합 | 모듈 간 연동 | 통합테스트 결과서 | 테스트 | 개발+QA |
| 시스템/성능 | 전체 시스템 | 성능 측정 보고서 | 테스트 | QA+인프라 |
| 사용자 인수 | 업무 시나리오 | 인수테스트 결과서 | 운영 유사 | 고객+PM |
| 병행 | 신구 시스템 비교 | 병행테스트 결과서 | 운영 | 전체 |

### 테스트 진행 로드맵

프로젝트 마일스톤에 맞춰 각 테스트 유형의 시작/종료 시점을 명시한다. 단위→통합→시스템→인수→병행 순서가 기본이며, 각 단계별 **진입 기준(Entry Criteria)**과 **완료 기준(Exit Criteria)**을 정의.

### 성능 테스트 수행 방안

- 성능 측정 대상 및 목표치 명시 (응답시간, TPS, 동시접속 등)
- 테스트 도구 및 수행 절차
- 병목 발견 시 튜닝 → 재측정 반복 절차
- TTA 인증 등 외부 검증 연계 시 명시

## 이행 방안

### 이행 3원칙

1. **사전 준비**: 통합/성능 테스트 완료 후 이행 시작
2. **리허설**: 본 이행 전 최소 1회 이행 리허설 수행
3. **롤백 계획**: 각 단계별 롤백 절차와 판단 기준 정의

### 이행 조직도 표준

```
총괄 PM
├── 이행 관리 (총괄 관리, 상황 운영, 대내외 커뮤니케이션)
├── 응용 개발 (시스템 분석/설계, 개발, 데이터 이관)
├── 아키텍처 (환경 점검, 설정 확인, 성능 검증)
└── 인프라 (H/W·S/W 셋업, 네트워크, Vendor 제품 설치)
```

### 비상 계획

장애 유형별 대응 방안과 복구 목표(RTO/RPO)를 표로 정리:

```
| 장애 유형 | 대응 방안 | RTO | RPO |
```

## 비용 산정

### 비용표 표준 구조

| 구분 | 항목 | 수량 | 단가 | 금액 | 비고 |
|------|------|------|------|------|------|
| H/W | 서버, 스토리지, 네트워크 장비 | H/W 구성표와 일치 | | | |
| S/W | OS, DBMS, 미들웨어, 보안 S/W | S/W 구성표와 일치 | | | |
| 라이선스 | 솔루션 라이선스 | SA 라이선스 분석과 일치 | | | |
| 유지보수 | 연간 유지보수 | | 라이선스의 15~22% | | |
| 개발 | 인건비 (FP 기반 또는 M/M 기반) | | | | BA 기능점수 참조 |

### 수량 정합성 검증

모든 수량은 **단일 출처**에서 산출:
- H/W 수량 → H/W 구성표 (TA)
- S/W 수량 → S/W 구성표 (TA)
- 라이선스 수량 → 라이선스 분석 (SA)

비용표 수량 ≠ 구성표 수량은 **가장 빈번한 Critical 이슈**.

## 팀 간 의존성

TA 산출물의 의존 관계:

```
BA ──────────────▶ TA  (요구사항 → 인프라 규모)
DA ──────────────▶ TA  (DB 라이선스 → 비용)
SA ──────────┬──▶ TA  (H/W 요구사항 → 인프라)
             └──▶ TA  (라이선스 → 비용)
TA ──────────────▶ BA  (총 사업비 → 사업 개요)
```

- BA 요구사항과 SA 솔루션 스펙이 인프라 규모를 결정한다
- DA의 DBMS 선정과 SA의 라이선스가 비용에 반영된다
- TA에서 산출된 총 사업비는 BA 사업 개요에 역으로 반영된다

### 권한 체계

| 데이터 유형 | 권한 |
|-----------|------|
| 인프라 수량, 서버 사이징, 플랫폼 | **TA** |
| 요구사항, 프로세스 | BA |
| 데이터 모델, 마이그레이션 | DA |
| 제품 스펙, 인증, 모델번호 | SA |

비권한 SSOT가 권한 SSOT와 충돌하면, 비권한 쪽이 수정된다.

## Critic 검증 포인트 (TA)

| 검증 항목 | Critical 조건 |
|---------|-------------|
| 환경 분리 | 운영/개발/테스트 중 하나라도 정의 누락 |
| 수량 불일치 | H/W 구성표 ≠ 비용표 수량 |
| HA 누락 | 운영 환경 주요 서버 이중화 미정의 |
| 망분리 미준수 | 금융 망분리 기준 위배 |
| 프로토콜 누락 | 인터페이스에 프로토콜/보안 미명시 |
| 백업 미정의 | RPO/RTO 미명시 또는 테스트 계획 없음 |
| 임계값 미정의 | 모니터링 지표에 임계값 없음 |
